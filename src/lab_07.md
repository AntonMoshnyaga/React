# Практична робота №7: Робота з URL-параметрами

## Відповіді на контрольні запитання

1. **Поясніть принципову різницю між URL-параметрами шляху (Path Parameters, напр. `/users/42`) та параметрами запиту (Query Parameters, напр. `/users?sort=asc`). В яких випадках який підхід слід використовувати?**
   - **Path Parameters (`/:id`)**: Використовуються для ідентифікації конкретного ресурсу або розмежування ієрархічної структури сторінок (наприклад, сторінка профілю конкретного користувача, сторінка окремого посту). Вони є обов'язковою та невід'ємною частиною маршруту.
   - **Query Parameters (`?key=value`)**: Використовуються для змінення способу відображення, фільтрації пошуку або сортування вже існуючого списку ресурсів на цій самій сторінці. Їх варто використовувати, коли параметр є необов'язковим і допоміжним (наприклад пошук, пагінація сторінок, напрямок сортування).

2. **Що робить хук `useParams()` і який обов'язковий синтаксис необхідно використати в визначенні `<Route path="...">` для того, щоб цей хук запрацював?**
   Хук `useParams()` дозволяє витягти з URL-шляху всі параметри (ідентифікатори, слаги) після переходу на поточний компонент і перетворити їх на зручний JavaScript-об'єкт `const { parameterName } = useParams()`. 
   Для того щоб хук їх "спіймав", обов'язково потрібно задати маршрут із символ двокрапки `:` в атрибуті path компонента Route: `<Route path="/post/:postId" element={<PostDetails />} />`. У такому випадку `postId` стає ключем у хуку.

3. **Чому зберігання стану пошукового рядка (Search Bar) в URL (через `useSearchParams`) вважається архітектурно кращим рішенням, ніж зберігання його у локальному стані компонента (`useState`)? Опишіть переваги для кінцевого користувача.**
   - **Можливість ділитися (Deep Linking):** Якщо ви шукаєте телевізори Samsung за ціною від 500$, і це зберігається в URL `?q=samsung&min=500`, ви можете скопіювати посилання і відправити другові. З `useState` він побачить просто пусту сторінку "Всі товари".
   - **Навігація:** Браузер фіксує зміну URL у своїй історії (History API). Це дозволяє кнопці "Назад" у браузері повертатися до попереднього стану пошуку.
   - **Закладки (Bookmarks):** Можна зберігати поточний стан фільтрів чи сторінку пошуку в закладки браузера.

4. **Якщо ви використовуєте метод масиву `filter()` прямо перед поверненням JSX-розмітки, чи відбувається ця фільтрація при кожному рендері компонента? Як це може вплинути на продуктивність для дуже великих масивів (напр. 10 000+ елементів) і який інший React-хук ви б використали для оптимізації?**
   - Так, фільтрація та сортування, написані в тілі компонента прямо перед `return`, виконуються *знову і знову* при кожному рендері (наприклад, коли користувач просто вводить символ в інпут або ховає/відкриває модальне вікно).
   - На масивах 10,000+ це спричинить помітні затримки інтерфейсу та фрізи (lag).
   - Оптимізувати це можна за допомогою хука `useMemo()`: `const filtered = useMemo(() => data.filter(...), [data, query])`. Завдяки `useMemo`, важка операція фільтрації виконається лише тоді, коли дані масиву або пошуковий запит зміняться. Якщо ж перемальовується не пов'язаний інпут — масив візьметься з кешу.

5. **В яких об'єктах стандарту Web API у звичайному (базовому) JavaScript зберігаються параметри після знаку питання `?`, і як React Router спрощує роботу з ними?**
   - У нативному JS (Web API) вони є частиною рядка `window.location.search`. Щоб розпарсити їх комфортно, використовується конструктор `new URLSearchParams(window.location.search)`.
   - React Router надає абстракцію — хук `useSearchParams()`. Він бере на себе всю логіку з парсингу, повертає об'єкт класу `URLSearchParams` і одразу дає функцію `setSearchParams()`, яка не лише оновлює URL, а й інтегрується з життєвим циклом React (автоматично викликає правильний рендер при зміні параметрів і пов'язана з історією переходів `history API`).
