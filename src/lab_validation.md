# Практична робота: Валідація введених даних

## Відповіді на контрольні запитання

1. **У чому головна різниця між керованими (controlled) та некерованими (uncontrolled) компонентами?**
   Керовані компоненти — це компоненти, стан яких "знаходиться в руках" React (`useState`). Контроль над значенням поля вводу (його `value` та обробка через `onChange`) здійснює React, і він є "єдиним джерелом істини". Некеровані компоненти зберігають свій стан безпосередньо у DOM за замовчанням, а React отримує доступ до елементів за допомогою референцій (хук `useRef`). Керовані забезпечують моментальну валідацію, а некеровані корисні тоді, коли форма дуже проста і перевірка потрібна лише при submit або коли потрібна інтеграція з не-React бібліотеками.

2. **Навіщо використовувати `useRef` при роботі з формами і в яких випадках це виправдано?**
   Хук `useRef` використовується для створення "посилання" (reference) на реальний DOM-елемент форми у некерованих компонентах. За допомогою `ref.current.value` можна отримати дані з інпута під час події типу Submit. Це виправдано у випадках, якщо форма величезна (сотні полів), і немає необхідності постійно перемальовувати компонент при кожному натисканні клавіші, а достатньо зібрати всі дані однією пачкою в момент відправлення. Або ж, наприклад, для керування фокусом (`inputRef.current.focus()`).

3. **Чому важливо викликати `event.preventDefault()` при обробці події `submit`?**
   Стандартна поведінка браузера при спрацьовуванні події `submit` форми полягає у перезавантаженні сторінки або надсиланні синхронного HTTP-запиту за адресою атрибута `action`. У React додатках (SPA), перезавантаження сторінки знищить поточний стан програми (state) і змусить завантажувати React та збирати DOM із нуля. Використання `event.preventDefault()` скасовує цю стандартну поведінку DOM, дозволяючи нам самим обробити інформацію через JavaScript і відправити дані на сервер асинхронно через `fetch` чи `axios`.
