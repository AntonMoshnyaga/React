# Практичне заняття №6: Налагодження коду (Debugging)

## Відповіді на контрольні запитання

1. **В чому полягає критична перевага використання React Developer Tools (Components) порівняно зі звичайним інспектуванням DOM-елементів (Elements) при пошуку помилок у React-додатку?**
   Звичайний інспектор DOM показує лише кінцевий HTML-результат, згенерований браузером. Він нічого не знає про компоненти React, їхній стан (State), пропси (Props) чи хуки. React Developer Tools (вкладка Components) дозволяє бачити віртуальне дерево (Virtual DOM), компонентну структуру, що ієрархічно відображає архітектуру додатка. Це дає можливість читати та змінювати "на льоту" стан і пропси конкретного компонента, щоб миттєво побачити, як UI реагує на реальні дані, уникаючи розстановки `console.log()` по всьому коду.

2. **Як відловити ситуацію, коли React виконує неоптимальний рендеринг списків через неправильне використання атрибута key?**
   По-перше, проаналізувавши стандартну вкладку Console — React відразу видасть попередження: `Warning: Encountered two children with the same key`. 
   По-друге, увімкнувши в налаштуваннях React DevTools (General) функцію **"Highlight updates when components render"**. Якщо при додаванні одного елемента у список підсвічуються (перемальовуються) абсолютно всі елементи, це означає, що React втратив їхню ідентичність через відсутні або дубльовані ключі (`key`) і руйнує/створює DOM-вузли з нуля.

3. **Поясніть алгоритм дій розробника при встановленні та використанні Breakpoint у вкладці Sources браузера Chrome. Що відбувається з виконанням програми в цей момент?**
   - Відкрийте панель розробника (F12) і перейдіть на вкладку **Sources**.
   - Натисніть `Ctrl/Cmd + P` і введіть ім'я файлу (наприклад, `Feed.jsx`).
   - Натисніть на номер рядка ліворуч від коду (з'явиться блакитна мітка — *Breakpoint*).
   - Виконайте на сторінці дію, яка спровокує виконання цього коду (клік, ввід тексту).
   У цей момент **виконання програми повністю призупиняється** (Paused in debugger). Ви можете наводити курсором на змінні, щоб побачити їхні поточні значення в пам'яті, досліджувати стек викликів (Call Stack) та область видимості (Scope).

4. **Яке ключове слово (резервоване в JavaScript) можна написати безпосередньо у вихідному коді компонента, щоб змусити браузер автоматично зупинити виконання при відкритій панелі розробника?**
   Це ключове слово `debugger;`. Якщо панель розробника (\DevTools) у браузері відкрита, виконання JavaScript автоматично зупиниться на цьому рядку, створивши точку зупинки без необхідності шукати потрібний файл вручну через вкладку Sources.

5. **Що означають дії "Step Over", "Step Into" та "Step Out" у панелі управління дебагером браузера (Sources)?**
   Ці інструменти дозволяють керувати покроковим виконанням призупиненого коду:
   - **Step Over (F10):** Виконати поточний рядок без заглиблення у виклики функцій на ньому. (Виконає цілу функцію відразу і перейде на наступний рядок поточного блоку).
   - **Step Into (F11):** Увійти *всередину* функції, яка викликається на поточному рядку, щоб продебажити її логіку зсередини.
   - **Step Out (Shift+F11):** Завершити виконання поточної функції і повернутися до того місця, звідки ця функція була викликана ("вистрибнути" назад).
