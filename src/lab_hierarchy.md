# Практичне заняття: Побудова ієрархії компонентів (Декомпозиція)

## Відповіді на контрольні запитання

1. **Які основні відмінності між презентаційним (Dumb) та контейнерним (Smart) компонентами у React?**
   - **Презентаційні (Dumb) компоненти:** Відповідають за те, *як речі виглядають* (UI). Вони зазвичай не мають власного стану (хіба що стан UI, наприклад, відкритий/закритий дропдаун), не роблять запитів до мережі і отримують усі дані та колбеки виключно через `props`. (Наприклад: `Button`, `UserInfo`, `ActivityItem`).
   - **Контейнерні (Smart) компоненти:** Відповідають за те, *як речі працюють* (логіка). Вони керують станом (State), "спілкуються" з Redux або Context, роблять запити на сервер (fetching data) і передають готові дані презентаційним компонентам вниз по ієрархії. (Наприклад: `ProfilePage`, `ProductCard`).

2. **Поясніть принцип "Джерело істини" (Single Source of Truth) при побудові компонентної ієрархії. Де зазвичай має зберігатися спільний стан (State)?**
   "Джерело істини" означає, що певний шматочок даних у вашому додатку повинен бути збережений лише в *одному* місці. Якщо кілька компонентів потребують тих самих даних, стан має зберігатися у найближчому спільному предку (parent) для цих компонентів (Lifting State Up). Спроба дублювати стан у кількох дочірніх компонентах призведе до помилок синхронізації та багів.

3. **Чому функція ітерації по масиву (`.map()`) зазвичай знаходиться на рівні проміжних композиційних компонентів (наприклад, `ActivityList`), а не всередині атомарних (наприклад, `ActivityItem`)?**
   Атомарний компонент (`ActivityItem`) проектується так, щоб візуалізувати лише *одну* сутність (один елемент списку). Його відповідальність — макет конкретної одиниці. Композиційний компонент (`ActivityList`) відповідає за структурну організацію списку (наприклад, додавання заголовка, рендер стану "список порожній", огортання у `<ul>`). Такий розподіл дозволяє легко перевикористовувати `ActivityItem` в інших частинах коду без прив'язки до логіки масиву.

4. **Що таке явище Prop Drilling і коли воно стає проблемою при глибокій ієрархії компонентів?**
   Prop Drilling — це процес передачі даних від компонента верхнього рівня до компонента далеко внизу дерева через низку "посередницьких" компонентів, яким ці дані самим по собі не потрібні. Це стає проблемою при високій вкладеності (наприклад, понад 3-4 рівні), коли змінити або додати нову властивість доводиться у 5 різних файлах одночасно. Щоб уникнути цього, застосовують Context API, Redux/Zustand або патерн Component Composition (`children`).

5. **Наведіть критерії, спираючись на які ви приймаєте рішення винести окремий шматок JSX-розмітки у новий незалежний файл компонента.**
   - **Перевикористання (Reusability):** Якщо шматок UI використовується у кількох місцях програми (наприклад, кнопки, інпути, картки товарів).
   - **Відповідальність (Separation of Concerns):** Якщо файл компонента стає занадто великим (наприклад, більше ніж 200-300 рядків) або бере на себе занадто багато різних функцій.
   - **Оптимізація рендерінгу:** Якщо частина UI часто перерендерюється (через складні анімації або локальний ввід тексту), але інша частина статична, динамічну частину краще винести в окремий компонент і застосувати до нього ізольований стан.
   - **Читабельність (Readability):** Щоб семантично назвати великий блок коду (наприклад, `<Header />`, замість того, щоб тримати довгий шматок `<div><nav>...` всередині `App`).
