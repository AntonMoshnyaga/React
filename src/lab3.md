## Фрагменти коду

### Логіка роботи пошуку і фільтрації
```javascript
const filteredPosts = postsData.filter(post => {
  const matchesSearch =
    post.content.toLowerCase().includes(searchTerm.toLowerCase()) ||
    post.author.toLowerCase().includes(searchTerm.toLowerCase());
  const matchesCategory = activeCategory === 'All' || post.category === activeCategory;
  return matchesSearch && matchesCategory;
});
```

### Пояснення, як реалізовано обробку "порожнього результату"
Обробка порожнього результату реалізована за допомогою тернарного оператора в JSX. Якщо довжина відфільтрованого масиву `filteredPosts.length > 0`, відображається список постів через метод `.map()`. Якщо ж відфільтрований масив порожній (умова хибна), відображається JSX елемент з повідомленням про те, що пости не знайдено: `<p className={styles.empty}>Нічого не знайдено за вашим запитом.</p>`.

## Відповіді на контрольні запитання

1. **Що таке "підняття стану" (lifting state up) і чому ми використовуємо його для `SearchBar`?**
   "Підняття стану" — це підхід у React, за якого стан переміщується до найближчого спільного предка компонентів, що його використовують. Для `SearchBar` ми підіймаємо `searchTerm` в компонент `App`, оскільки значення пошуку потрібне не лише `SearchBar` (для видимості в інпуті), але й самому `App` для фільтрації масиву даних `postsData` перед рендерингом.

2. **Поясніть асинхронну природу оновлення стану в `useState`.**
   Коли ми викликаємо функцію оновлення стану (наприклад, `setSearchTerm`), React не змінює значення миттєво. Натомість він планує оновлення стану і сигналізує про необхідність повторного рендеру компонента. Саме тому, якщо звернутись до змінної стану відразу після виклику функції оновлення, ми отримаємо попереднє значення.

3. **Чому для фільтрації ми створюємо нову змінну `filteredPosts`, а не змінюємо оригінальний масив `postsData` у стані?**
   Ми не змінюємо `postsData` напряму (мутація), оскільки, по-перше, не можна мутувати оригінальні (початкові) дані — вони потрібні нам, якщо користувач очистить пошук і ми знову захочемо показати всі результати. По-друге, похідні дані (`filteredPosts`), які можна обчислити під час рендеру з існуючого стану та властивостей, не повинні зберігатися у власному стані, щоб уникнути десинхронізації.

4. **У чому перевага використання керованих компонентів над некерованими при реалізації пошуку?**
   Керовані компоненти дозволяють тримати "джерело істини" у React-стані, а не у DOM. Це дає нам можливість взаємодіяти з даними синхронно з введенням користувача: обробляти значення на льоту, виконувати валідацію, оновлювати інші компоненти відразу і забезпечувати миттєву фільтрацію без необхідності звертатись до DOM-елементів (через `ref`).
